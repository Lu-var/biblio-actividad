Microservicios en Acción:
Gateway + Eureka

Vas a diseñar tu propio caso, levantar los requerimientos en 5 minutos y construir una arquitectura de microservicios real, con descubrimiento de servicios y un punto de entrada único.

Spring Cloud Gateway Netflix Eureka Mínimo 3 microservicios Tema libre Entrega vía GitHub + AVA

Contexto

Hasta ahora cada microservicio que has construido vive solo: tú sabes su puerto, tú lo llamas directo. Eso funciona con uno o dos servicios corriendo en tu máquina. No funciona en un sistema real, donde los servicios se levantan, se caen, se replican y cambian de IP constantemente.

!

El problema

Si el cliente tiene que conocer de antemano la URL exacta de cada microservicio, el sistema se vuelve frágil: cualquier cambio de puerto o IP rompe todo, y no hay forma simple de balancear carga entre instancias.

✓

La solución

Eureka actúa como una libreta de direcciones viva: cada microservicio se registra al levantar y se da de baja al caer. Spring Cloud Gateway es el único punto de entrada, que consulta esa libreta y enruta cada petición al servicio correcto.

Objetivos de aprendizaje

Al terminar esta actividad deberías poder responder, con tu propio proyecto como evidencia, las siguientes preguntas:

  • 01¿Cómo se levanta un Eureka Server y por qué los microservicios necesitan registrarse en él?
  • 02¿Cómo configura Spring Cloud Gateway las rutas hacia distintos microservicios usando el nombre del servicio en lugar de su IP?
  • 03¿Cómo se traduce un conjunto de requerimientos de negocio en una arquitectura de microservicios separados por responsabilidad?
  • 04¿Cómo se integran estos conceptos nuevos con lo que ya dominas: capas, DTOs, persistencia, Docker?

Dinámica de la clase

La actividad se desarrolla completa durante el bloque de clases, en dos momentos bien diferenciados:

1

Inventa tu caso y levanta requerimientos 5 min

Define de qué trata tu sistema, qué entidades maneja y qué necesita resolver cada microservicio. No es momento de codear: es momento de decidir el qué, no el cómo. Deja por escrito tus requerimientos, aunque sea en una lista simple — los vas a necesitar para docs/.

2

Implementa la arquitectura

Con los requerimientos definidos, construye tu Eureka Server, tu Gateway y al menos tres microservicios que le den vida al caso que inventaste. Aplica todo lo que ya sabes: capas, persistencia, configuración, control de versiones.

3

Documenta y sube tu trabajo

Arma el README.md de la raíz y la documentación en docs/, sube todo a GitHub y entrega el enlace del repositorio en el AVA antes del cierre de la clase.

Requisitos del proyecto

Caso de negocio

El tema es libre: tú lo inventas. Tiene que tener sentido como sistema dividido en al menos tres dominios de responsabilidad distintos. Algunas ideas, si necesitas un punto de partida:

Delivery de comida Reserva de canchas deportivas Tienda de tecnología Eventos universitarios Biblioteca Plataforma de cursos online Arriendo de equipamiento

Arquitectura mínima obligatoria

  • 1 Eureka Server funcionando como registro de servicios.
  • 1 API Gateway (Spring Cloud Gateway) como punto de entrada único, enrutando por nombre de servicio.
  • 3 o más microservicios, cada uno registrado en Eureka, con lógica de negocio real (no solo un "Hello World").
  • Cada microservicio expone al menos un endpoint REST coherente con el caso que inventaste.

Estructura del repositorio

El repositorio que entregues debe organizarse de forma clara. Como mínimo:

tu-proyecto/
├── README.md ← obligatorio en la raíz
├── docs/ ← documentación en markdown
│   ├── requerimientos.md
│   ├── arquitectura.md
│   └── api.md
├── eureka-server/
├── api-gateway/
├── microservicio-1/
├── microservicio-2/
└── microservicio-3/

¿Qué debe contener el README.md de la raíz?

  • Nombre y breve descripción del caso que inventaste.
  • Listado de los microservicios y qué resuelve cada uno.
  • Instrucciones para levantar el proyecto (orden de arranque: Eureka → microservicios → Gateway).
  • Puertos utilizados y, si corresponde, enlace a la carpeta docs/.

¿Qué va en docs/?

  • requerimientos.md: lo que levantaste en los primeros 5 minutos de la clase.
  • arquitectura.md: descripción de cada componente y cómo se comunican entre sí.
  • api.md: endpoints expuestos por cada microservicio, con método, ruta y un ejemplo de respuesta.
